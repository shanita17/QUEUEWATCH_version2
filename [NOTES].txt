1. index.html — Home Page
----------------------------------------------------------------
This is the main landing page and the first page users see when
they open the website. It displays a hero section with the
project tagline, a search bar to find branches, and a live
stats bar showing how many branches are quiet, busy, very busy,
or offline. Below that it shows all 31 real Gauteng branch
cards in a grid layout with filter tabs (All, Home Affairs,
SASSA, Clinic, Licensing).

Clicking any branch card saves that branch's data to
localStorage and navigates to branch.html to show its full
details. This page loads script.js which fetches branch data
from the real backend API at /api/branches.

STATUS: Fully connected to backend. Real data. ✅

----------------------------------------------------------------
2. branches.html — All Branches Page
----------------------------------------------------------------
A dedicated page for browsing all 31 real Gauteng branches
with more advanced filtering options. Has three filters: a
text search, a dropdown to filter by branch type (Home Affairs,
SASSA, Clinic, Licensing), and a dropdown to filter by current
status (Quiet, Busy, Very Busy, System Offline).

Loads script.js which fetches from /api/branches, then runs
its own filterAll() function to apply the extra dropdown
filters on top. Clicking a card works the same way as the
home page — saves to localStorage and goes to branch.html.

STATUS: Fully connected to backend. Real data. ✅

----------------------------------------------------------------
3. branch.html — Branch Detail Page
----------------------------------------------------------------
This page shows the full details of a specific branch after a
user clicks on a branch card. It reads the selected branch
from localStorage and displays the branch name, type, current
status badge, average wait time, and address.

On the left it shows a list of real reports fetched from
/api/reports/branch/:id — showing who reported, what time,
how long the wait was, and any notes they left. If no reports
exist, it shows a friendly message. The "+ Add Report" button
links to report.html.

On the right is a map placeholder (Google Maps API coming),
and a summary panel with total reports, average wait, and
last update time. All data is real and live from the database.

STATUS: Fully connected to backend. Real reports loading. ✅

4. auth.html — Login / Register Page
----------------------------------------------------------------
Authentication page where users can log in or register. Uses
two tabs that switch between forms without reloading.

Login sends a POST to /api/auth/login. Register sends a POST
to /api/auth/register. On success a session is created on
the server and the user is redirected to the home page.
Error messages from the API are shown as toast notifications.

5. report.html — Submit a Queue Report Page
----------------------------------------------------------------
Page where a logged-in user can submit a queue report. The
branch dropdown is dynamically loaded from /api/branches. If
the user came from a branch detail page, that branch is
pre-selected automatically.

Collects branch, status, estimated wait time, and optional
notes. Sends a POST to /api/reports. If the user is not
logged in, they are redirected to auth.html. On success the
report is saved to the database.

STATUS: Fully connected to backend. Saves to database. ✅

----------------------------------------------------------------
6. admin.html — Admin Dashboard Page
----------------------------------------------------------------
Protected admin page for moderating user-submitted reports.
When the page loads it calls /api/auth/me to verify the user
is logged in AND has the 'admin' role. If not, it shows a
toast and redirects to auth.html immediately.

Once authenticated, loads all reports from /api/reports and
displays them in a table. Flagged reports are highlighted
in red. The Remove button calls DELETE /api/reports/:id to
permanently delete the report from the database.

STATUS: Protected. Fully connected to backend. Real delete. ✅

11. script.js — Shared JavaScript File
----------------------------------------------------------------
The most important frontend JavaScript file. Loaded by
index.html and branches.html. Fetches all 31 real branch
data from /api/branches and handles rendering, filtering,
and navigation.

Key functions:
- loadBranches(): fetches from /api/branches and maps API
  response to frontend format
- waitLabel(w): converts minutes to "2 hrs wait" etc.
- timeAgo(dateStr): converts timestamp to "5 min ago" etc.
- renderCards(list): builds branch cards with onclick that
  saves branch to localStorage and goes to branch.html
- updateStats(list): updates quiet/busy/very busy/offline
  chip counters
- getFiltered(): filters branches by search and active tab
- filterCards(): calls getFiltered, renderCards, updateStats
- setFilter(type, el): updates active tab and re-renders
- showToast(msg): shows bottom notification for 3 seconds
- Safety guard: only calls loadBranches() on pages with
  a branchGrid element

STATUS: Fully connected to /api/branches. ✅

SERVER FILES (inside /server folder)

----------------------------------------------------------------
13. server.js — Express Application Entry Point
----------------------------------------------------------------
Main Node.js file that starts the web server on port 3000.
Sets up Express with CORS, JSON parsing, and session
middleware. Serves frontend files statically from the parent
folder. Mounts three API route files under /api. Includes
a /api/health check endpoint.

Run with: node server.js (from inside the /server folder)

----------------------------------------------------------------
14. db.js — Database Connection
----------------------------------------------------------------
Creates and exports a shared SQL Server connection pool using
mssql@10.0.4. Reads config from .env. Uses SQL Server
Authentication. Logs success or exits on failure.

IMPORTANT: Do NOT upgrade mssql — newer versions break on
this machine. TCP/IP is disabled on Shanita; this config
works around that using SQL Server Auth on the default port.

----------------------------------------------------------------
15. routes/branches.js — Branches API Routes
----------------------------------------------------------------
GET / — returns all 31 branches with live stats calculated
via SQL subqueries: latest status today, avg wait today,
total reports today, and time of last report.
GET /:id — returns a single branch by BranchID.

----------------------------------------------------------------
16. routes/reports.js — Reports API Routes
----------------------------------------------------------------
GET /branch/:id — all reports for a branch, newest first,
joined with username. No auth required.
GET / — all reports for admin dashboard. Admin only.
POST / — create a new report. Requires active session.
DELETE /:id — delete a report. Admin only.
PATCH /:id/flag — flag a report. Admin only.

----------------------------------------------------------------
17. routes/auth.js — Authentication API Routes
----------------------------------------------------------------
POST /register — hashes password with bcryptjs, inserts
user, starts session.
POST /login — checks email, compares password (handles both
hashed and plain text for seed data), creates session.
POST /logout — destroys session.
GET /me — returns current logged in user or 401.
