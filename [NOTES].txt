PROJECT SUMMARY
----------------------------------------------------------------
QueueWatch SA is a full-stack community queue-reporting web
app for South African public service offices (Home Affairs,
SASSA, Clinics, Licensing). Citizens report real-time queue
conditions at registered Gauteng branches so others can plan
smarter before they leave home.

Stack: Node.js + Express.js backend, SQLite database
(better-sqlite3), Vanilla HTML/CSS/JS frontend, Brevo
transactional email API, session-based auth (bcryptjs),
deployed on Render.com.

3. branch.html — Branch Detail Page

4. auth.html — Login / Register / Forgot Password Page
----------------------------------------------------------------
Single page with three tab states: Login, Register, Forgot
Password. Tabs switch without page reload.

Login: POST /api/auth/login → session cookie → redirect home
Register: POST /api/auth/register → session → redirect home
Forgot: POST /api/auth/forgot-password → sends reset email

CAPTCHA is required on the Register form. Generated by
GET /api/auth/captcha (SVG image). A refresh button lets
the user get a new code. CAPTCHA answer is validated
server-side against req.session.captcha.

Password rules enforced both client and server side:
- Min 8 chars
- At least one uppercase, one lowercase, one digit,
  one special character

5. reset.html — Password Reset Page
----------------------------------------------------------------
Reached via a link emailed to the user (from forgot password
flow). Reads a base64-encoded token from the URL query param
(?token=...). User sets a new password (same rules as
register). Sends POST /api/auth/reset-password with the
token and new password.

Note: The token is a simple Base64 encoding of the user's
email — this is a prototype implementation. Production would
use a time-limited signed token (e.g. JWT or UUID stored in
DB with expiry).

Loads: (no shared JS, standalone)
API calls: POST /api/auth/reset-password

STATUS: Fully connected to backend. ✅

6. report.html — Submit a Queue Report Page

7. admin.html — Admin Dashboard

Loads: nav-auth.js

12. script.js — Core Frontend Controller
----------------------------------------------------------------
Shared JS loaded by index.html and branches.html. Fetches
all branch data and handles all card rendering and filtering.

Key functions:
- loadBranches(): GET /api/branches, maps API fields to
  frontend object shape {id, name, type, address, lat, lng,
  status, wait, reports, updated}
- waitLabel(w): converts minutes → "30 min wait" / "2 hrs wait"
- timeAgo(dateStr): timestamp → "5 min ago" / "2 hrs ago"
- renderCards(list): builds branch cards with onclick that
  saves to localStorage and navigates to branch.html
- updateStats(list): updates Quiet/Busy/Very Busy/Offline counters
- filterCards(): combines filter + search, calls renderCards
- setFilter(type, el): tab filter, updates active class
- showToast(msg): toast notification, auto-hides after 3s
- Guard: only runs loadBranches() on pages with #branchGrid

STATUS: Fully connected. ✅

----------------------------------------------------------------
13. nav-auth.js — Dynamic Navigation Controller
----------------------------------------------------------------
Loaded on every page. On DOMContentLoaded, calls
GET /api/auth/me to check login state.

If NOT logged in: hides "Admin" nav link, shows Login +
Register buttons (default HTML state).

If logged in as regular user: hides "Admin" link, replaces
Login/Register buttons with "{Full Name}" label + Logout.

If logged in as admin: shows ONLY "Admin" link (hides all
others), shows "ADMIN" yellow badge + Logout button.

handleGlobalLogout(): POST /api/auth/logout → redirect home.

STATUS: Fully connected. Role-based nav working. ✅

17. db.js — SQLite Database Connection
----------------------------------------------------------------
Opens a single SQLite database file at /server/database.sqlite
using the sqlite + sqlite3 packages. Creates all three tables
on startup if they don't exist (CREATE TABLE IF NOT EXISTS).
Seeds Branches, Users, and Reports if Branches table is empty.

Tables:
  Users (UserID, FullName, Email, Password, Role, CreatedAt)
  Branches (BranchID, Name, Type, Address, Latitude,
            Longitude, CreatedAt)
  WaitingTimeReports (ReportID, BranchID, UserID, Status,
                      WaitMinutes, Notes, IsFlagged, CreatedAt)

Seed data: 8 Gauteng branches, 1 admin user, 1 test user,
8 sample reports. Admin and test user passwords are stored
as plain text in seed data (bcrypt comparison handles this
via the startsWith('$2') check in auth.js login route).

SECURITY NOTE: Production seed data should use pre-hashed
passwords. Plain text passwords in seed are prototype-only.

Exports: { poolPromise }

18. routes/auth.js — Authentication Routes
----------------------------------------------------------------
All routes mounted under /api/auth.

POST /register
  - Validates CAPTCHA (req.session.captcha)
  - Validates password regex (uppercase, lowercase, digit,
    special char, min 8)
  - Checks email uniqueness
  - bcrypt.hash(password, 10)
  - Inserts user, starts session, sends welcome email async

POST /login
  - Fetches user by email
  - bcrypt.compare for hashed passwords; plain === for seeds
  - Sets req.session.user (UserID, FullName, Email, Role)

POST /logout
  - req.session.destroy()

GET /me
  - Returns req.session.user or 401

PUT /profile
  - Requires session. Updates FullName + Email. Optionally
    updates password if newPassword provided and passes regex.
  - Checks email uniqueness against other users.

POST /forgot-password
  - Looks up user by email (always returns 200 for security)
  - Generates base64(email) as a prototype reset token
  - Sends reset link email via Brevo

POST /reset-password
  - Decodes base64 token → email
  - Validates password regex
  - bcrypt.hash and updates DB

POST /contact
  - Validates CAPTCHA
  - Sends contact form email to admin via Brevo

GET /captcha
  - Generates a 5-char alphanumeric SVG CAPTCHA
  - Stores lowercased answer in req.session.captcha
  - Returns SVG image (no-cache headers)

Email sending: sendAppEmail() uses Brevo HTTP API directly
(not nodemailer SMTP) to avoid Render's SMTP port block.
Falls back to console.log if BREVO_API_KEY not set (dev mode).

STATUS: ✅

----------------------------------------------------------------
19. routes/branches.js — Branch Routes
----------------------------------------------------------------
All routes mounted under /api/branches.

GET /
  - Returns all branches with live computed stats via SQL
    subqueries: Status (latest today), AvgWait (today),
    ReportsToday (count today), LastReportAt (most recent ever)
  - Ordered alphabetically by Name

GET /:id
  - Returns single branch by BranchID

NOTE: No POST, PUT, or DELETE routes exist. Admin branch
management (add/edit/deactivate) is NOT yet implemented.
This is a known gap for future development.

STATUS: Read-only. ✅

----------------------------------------------------------------
20. routes/reports.js — Report Routes
----------------------------------------------------------------
All routes mounted under /api/reports.

GET /branch/:id
  - Public. All reports for a branch, newest first.
  - Joins Users to get UserName.

GET /user
  - Requires session. Returns all reports by the logged-in
    user. Joins Branches to get BranchName. Includes
    IsFlagged for accuracy rate calculation.

GET /stats
  - Admin only. Returns { users: N, branches: N } counts.

GET /
  - Admin only (no explicit auth check — relies on admin
    page to guard access). Returns all reports with UserName
    and BranchName joined.

POST /
  - Requires session. Validates branchId, status (one of 4
    valid values), optional waitMinutes and notes.

DELETE /:id
  - Admin only. Hard deletes report from DB.

PATCH /:id/flag
  - Admin only. Sets IsFlagged = 1 on report.

KNOWN GAPS / TODO
================================================================

1. Admin branch management UI — branches.js has no write
   routes. Admin cannot add/edit/deactivate branches via
   the app. Must be done directly in the database.

2. Password reset token security — current token is just
   base64(email). No expiry, no single-use enforcement.
   Acceptable for prototype; needs a UUID+expiry in DB
   for production.

3. GET /api/reports has no admin auth guard server-side —
   currently relies on the admin.html page to check role
   before displaying. Should add middleware protection.

4. database.sqlite is not committed to git (.gitignore).
   On fresh Render deploy, the database is re-seeded from
   scratch. Persistent disk storage needed for production.

5. 31 Gauteng branches referenced in NOTES and docs, but
   only 8 are seeded in db.js. The rest were inserted
   directly in an earlier version. Check the live database
   to confirm actual branch count.
